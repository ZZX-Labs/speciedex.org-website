Open Biodiversity Protocol Interface

SpeciedexAPI

SpeciedexAPI is the open, versioned, blockchain-aware application programming interface for querying, validating, submitting, streaming, exporting, and integrating data from the Speciedex ecosystem.

It provides a common machine interface to species blockchains, taxonomy, observations, conservation information, geography, ecological relationships, media, scientific references, validation state, peer-network status, and Bitcoin-funded data bounties.

SpeciedexAPI may be served by a local SpeciedexCore node, a full-weight server node, an institutional deployment, a regional node, or a public SpeciedexWeb gateway.

Homo sapiens is permanently excluded from all SpeciedexAPI species, observation, biometric, genomic, tracking, and profiling interfaces.

What SpeciedexAPI Is

SpeciedexAPI is the programmable service layer of the Speciedex protocol.

It allows software to inspect current species records, traverse historical chain state, search shared indexes, validate proposed submissions, monitor the live validation feed, retrieve source provenance, export datasets, and interact with approved contribution workflows.

The API exposes verified protocol state without requiring each client application to implement storage, peer synchronization, indexing, or blockchain validation independently.

Why a Dedicated API Exists

Speciedex is intended to support websites, applications, research tools, command-line clients, field systems, institutional software, analytical notebooks, artificial intelligence systems, and independent scientific services.

A stable and documented API allows these systems to use the same underlying verified dataset while remaining independently designed and maintained.

SpeciedexAPI separates protocol data access from any single graphical interface or deployment environment.

Interface Principles

SpeciedexAPI Design Principles

Versioned

Public interfaces use explicit versions so clients can migrate deliberately rather than break without warning.

Provenance-Preserving

Responses retain source, chain, block, object, validation, confidence, license, and revision information where applicable.

Blockchain-Aware

Clients may query current canonical state or inspect historical state at a specified species-chain position.

Implementation-Neutral

The interface is defined independently from a particular programming language, database, web framework, or client.

Machine-Readable

Stable field names, documented schemas, typed values, and predictable error objects support automated systems.

Human-Inspectable

Responses remain understandable through ordinary tools, browser documentation, command-line clients, and readable serialized formats.

Least Authority

Read, validation, submission, administration, publishing, and Bitcoin operations remain separated by scope.

Reproducible

Stable identifiers, pagination rules, snapshots, chain positions, and query metadata support repeatable results.

Access Models

Local, Institutional, and Public Access

Local Node API

A user-owned SpeciedexCore node may expose the API only to the local machine, a trusted local network, or an authenticated private client.

This mode supports private analysis, offline research, local-speed access, controlled submissions, and direct verification against user-owned chain storage.

Full-Weight Server API

A full-weight node can expose complete read, query, validation, feed, export, synchronization, and approved submission services over the complete Speciedex corpus.

Institutional API

Museums, universities, laboratories, conservation organizations, governments, and archives may expose Speciedex-compatible interfaces over public data and clearly separated local collections.

Public Gateway API

SpeciedexWeb gateways provide rate-limited public access for users and applications that cannot or do not wish to operate a node.

Public gateways identify their serving node, software version, synchronization state, release, limitations, and supported API version where practical.

Read Interfaces

Search, Retrieve, Traverse, and Compare

Read interfaces expose both current canonical records and historical protocol state.

Clients may retrieve one record by stable identifier, search across indexed fields, traverse linked entities, request selected representations, compare records, or reconstruct a species profile at a selected blockchain height or timestamp.

Responses distinguish canonical summaries from source assertions, disputed claims, provisional material, restricted fields, and derived indexes.

Stable Identifiers

Resources are addressed using durable Speciedex identifiers independent of display names and taxonomic changes.

Historical State

Supported queries may identify chain height, block, release, checkpoint, or date to reconstruct earlier state.

Field Selection

Clients may request selected fields, related entities, or compact representations where supported.

Relationship Expansion

Responses may expand taxonomy, geography, sources, observations, references, evidence, and ecological links within documented limits.

Structured Biodiversity Queries

Text Search

Search names, synonyms, common names, descriptions, identifiers, references, locations, and indexed text.

Taxonomic Query

Filter by domain, kingdom, phylum, class, order, family, genus, species, clade, rank, lineage, or taxonomic concept.

Geospatial Query

Search permitted data by coordinates, bounding areas, regions, habitats, protected areas, elevation, or depth.

Temporal Query

Filter observations, publications, assessments, blocks, revisions, and validation events by time.

Evidence Query

Filter by photograph, audio, video, specimen, literature, environmental DNA, sensor, or other evidence class.

Provenance Query

Chain Query

Filter by species chain, block range, entry type, checkpoint, schema version, commit state, or update age.

Data-Quality Query

Locate incomplete, stale, disputed, weakly evidenced, duplicated, conflicting, or bounty-eligible records.

Validation Services

Validate Before Submission

Validation endpoints allow clients to test proposed records without immediately submitting them to the network.

A validation response may report passed checks, failed checks, warnings, unresolved references, possible duplicates, taxonomic conflicts, geographic inconsistencies, missing evidence, unsupported licenses, prohibited content, and required schema upgrades.

Validation results do not guarantee final acceptance. Network policy, expert review, conflict resolution, evidence availability, challenge periods, and consensus requirements may still apply.

Schema

Required fields, value types, units, encodings, enumerations, identifiers, and version compatibility.

Taxonomy

Name, rank, lineage, authority, synonym, concept, proposed taxon, and classification consistency.

Evidence

Object hashes, media, references, methods, specimen records, timestamps, and evidentiary sufficiency.

Geography

Coordinates, uncertainty, regions, habitat plausibility, restricted locations, elevation, depth, and movement.

Provenance

Source identity, acquisition, license, attribution, signatures, transformations, and duplicate lineage.

Policy

Human exclusion, malicious payloads, prohibited records, unsupported content, and protocol-level restrictions.

Conflicts

Contradictory assertions, competing taxonomies, duplicate records, incompatible identifiers, and unresolved claims.

Chain Commit

Parent linkage, signature validity, schema support, object availability, ordering, and canonical-state checks.

Write Interfaces

Controlled Submission Workflows

Write interfaces are more restricted than public read interfaces.

A client may prepare and submit observations, corrections, references, media metadata, taxonomic proposals, source imports, validation responses, or bounty results only through authenticated and policy-compliant workflows.

Submission does not directly modify a canonical species blockchain. Proposed data enters staging, validation, review, conflict analysis, and consensus processing before acceptance.

Signed Submissions

Supported write requests use authenticated identities, scoped credentials, request signatures, or signed offline packages.

Idempotency

Submission identifiers and idempotency controls prevent accidental duplication during retries.

Staging Receipts

Accepted requests receive stable submission identifiers, validation state, timestamps, and subsequent status links.

Chain Receipts

Successfully committed work can return the species chain, block or entry identifier, canonical status, and propagation state.

Streaming Interfaces

Validation Feed and Live Events

SpeciedexAPI exposes the ticker-like Speciedex validation feed through documented streaming interfaces.

Clients can follow incoming submissions, validation stages, accepted chain entries, rejections, conflicts, revisions, bounty events, peer propagation, synchronization state, releases, and node-health alerts.

Streams may be filtered by species, taxon, region, provider, event type, validation state, bounty, severity, chain, or checkpoint.

Server-Sent Events

Lightweight one-way event streams for websites, dashboards, monitoring, and public feeds.

WebSocket

Bidirectional live sessions for interactive applications, operator consoles, and analytical interfaces.

JSON Lines

Append-friendly event streams for command-line tools, archives, pipelines, and deterministic replay.

Local Sockets

Node-local feeds for trusted software operating on the same system or private network.

Reliable Query Behavior

Pagination, Ordering, and Consistency

Cursor Pagination

Large and changing collections should use opaque cursors tied to a documented query state rather than fragile page numbers.

Deterministic Ordering

Every paginated result defines stable ordering and a deterministic tie-breaker.

Snapshot Queries

Clients may bind a multi-page query to a release, checkpoint, chain state, or server-generated snapshot token.

Fresh Queries

Live queries may intentionally include newly accepted data and clearly report that the result set can change during traversal.

Client Reliability

Errors, Status, and Diagnostics

API errors use stable machine-readable codes together with human-readable explanations.

Error responses may include request identifiers, validation paths, retry guidance, supported versions, rate-limit state, missing dependencies, rejected fields, conflict identifiers, and relevant documentation links.

Sensitive internal details, private credentials, raw exceptions, database paths, wallet information, and security controls must not be exposed through public error responses.

Client Errors

Invalid syntax, unsupported fields, malformed identifiers, incompatible versions, or failed authentication.

Validation Errors

Structured failures identifying schema, taxonomy, evidence, provenance, geography, policy, or chain issues.

Conflict Responses

Duplicate submissions, stale state, incompatible edits, unresolved scientific conflicts, or chain-tip changes.

Service Status

Temporary unavailability, synchronization lag, dependency failure, maintenance, capacity, or unsupported operation.

Authentication and Authorization

Scoped Access and Least Privilege

Public biodiversity reads may be anonymous where an operator chooses to provide them.

High-volume access, private institutional resources, validation workflows, submissions, reviewer operations, publishing, administration, synchronization control, and Bitcoin functions require appropriate authentication and narrowly scoped authorization.

Read Scope

Search and retrieve permitted public records and metadata.

Validate Scope

Submit proposed records for non-committing validation.

Submit Scope

Enter signed records into staged review and consensus workflows.

Review Scope

Perform authorized scientific, institutional, or policy review actions.

Publish Scope

Approve manifests, releases, schemas, or other controlled protocol artifacts where authorized.

Administrative Scope

Manage node-local configuration without automatically granting scientific publication or wallet authority.

Bitcoin-Native Services

Bounties and Settlement Metadata

SpeciedexAPI can expose public data-bounty definitions, eligibility rules, required evidence, validation state, reward amount, funding status, deadlines, and settlement metadata.

Eligible accepted contributions may be compensated using Bitcoin on-chain or through the Lightning Network.

Wallet seeds, private keys, macaroon-equivalent secrets, signing authority, internal invoices, and custodial controls must never be exposed through ordinary public API responses.

Speciedex does not issue or require a native token.

Security

API Security Model

Input Validation

Validate route parameters, query fields, bodies, uploads, object identifiers, encodings, and serialized structures.

Rate Controls

Apply documented limits, quotas, backpressure, request size constraints, and abuse controls to public gateways.

Transport Security

Public remote interfaces should use authenticated encryption and current deployment security practices.

Request Signing

Sensitive write, publication, synchronization, and administrative operations may require signed requests.

Audit Logging

Record security-relevant and state-changing operations while minimizing unnecessary personal information.

Object Quarantine

Untrusted uploads remain isolated until integrity, content, policy, and malware-related checks succeed.

Secret Separation

API servers, validators, publishers, administrators, and Bitcoin wallets should not share unnecessary credentials.

Safe Failure

Invalid input, incomplete dependencies, synchronization lag, and internal failure must not corrupt verified local state.

Privacy and Conservation

Restricted and Sensitive Information

Sensitive Species Locations

Coordinates and field details that could facilitate poaching, illegal collection, trafficking, nesting-site disturbance, habitat damage, or exploitation may be generalized, delayed, withheld, or restricted.

Minimal Contributor Disclosure

Public responses should expose only the contributor, reviewer, institutional, authorship, or payout metadata required for provenance and protocol transparency.

Human Exclusion

The API rejects human species records, human observations, human biometric data, facial-recognition records, human genomic profiles, person tracking, and population surveillance datasets.

Contextual Human References

Humans may appear only as necessary contextual entities associated with authorship, institutions, provenance, conservation activity, licensing, or interactions involving permitted non-human species.

Compatibility

Versioning and Deprecation

API versions are distinct from Speciedex protocol versions, SpeciedexCore software versions, schema versions, and dataset releases.

Compatible additions may be introduced without changing a major API version when clients are expected to ignore unknown fields.

Breaking changes require a new version, migration documentation, compatibility notes, and a reasonable deprecation period where practical.

API Version

Defines routes, request forms, response structures, errors, pagination, and client-visible behavior.

Protocol Version

Defines valid species chains, entries, consensus, signatures, policy, synchronization, and constitutional rules.

Schema Version

Defines structured records, fields, data types, constraints, and validation semantics.

Dataset Release

Identifies a reproducible public snapshot, manifest, chain state, object inventory, and associated indexes.

Scientific Reproducibility

Citable and Repeatable Results

A reproducible API response should identify the API version, protocol version, schema version, serving node or gateway, dataset release or chain state, query parameters, ordering, pagination state, and generation time where applicable.

Researchers should be able to preserve a query definition, cite its resulting chain state, export the permitted records, and repeat the analysis against the same historical snapshot.

Live queries should clearly indicate that newer accepted data may produce different results.

Project Status

An Evolving Public Interface

SpeciedexAPI remains part of the evolving Speciedex research and development architecture.

Documentation may describe conceptual, experimental, prototype, development, testing, release-candidate, or production interfaces.

Implementations must identify their supported API, protocol, schema, and dataset versions rather than implying that all documented capabilities are currently deployed.
