Biodiversity Data Infrastructure

SpeciedexCore

SpeciedexCore is the open-source data, taxonomy, indexing, validation, search, synchronization, and application engine that powers the Speciedex biodiversity platform.

It is designed to ingest biological records from many independent sources, preserve their provenance, reconcile conflicting classifications, normalize data into structured models, expose documented interfaces, and make biodiversity information searchable across species, geography, ecology, media, observations, conservation, research, and taxonomy.

SpeciedexCore is intended to support public websites, scientific tools, offline archives, institutional mirrors, research systems, educational applications, field devices, APIs, and independent biodiversity databases.

What SpeciedexCore Is

SpeciedexCore is the backend foundation of the Speciedex project ecosystem.

It provides the structured services required to store, connect, validate, search, update, export, and preserve biodiversity records.

Rather than treating biodiversity information as a collection of disconnected pages, SpeciedexCore models organisms, names, places, habitats, observations, references, specimens, media, genetic identifiers, ecological relationships, and conservation records as linked data.

Why a Dedicated Core Exists

Biodiversity data arrives from sources with different identifiers, licenses, taxonomic opinions, date formats, geographic systems, confidence levels, naming conventions, and technical structures.

A dedicated core system is required to preserve the source data while creating a usable common structure.

SpeciedexCore separates durable scientific records and data-processing logic from any single website, graphical interface, mobile application, search page, or deployment environment.

Engineering Principles

Core Design Goals

Source Traceability

Every imported or generated record should retain sufficient provenance to identify its source, retrieval date, license, original identifier, and transformation history.

Scientific Transparency

Conflicting names, uncertain identifications, disputed classifications, provisional records, and incomplete evidence should remain visible.

Interoperability

Data should be available through documented formats, stable identifiers, APIs, exports, and open standards.

Modularity

Importers, validators, search engines, storage adapters, media systems, maps, and application services should remain independently maintainable.

Resilience

The system should support mirrors, backups, offline copies, reproducible builds, independent deployments, and recovery after infrastructure failure.

Scalability

The architecture should support millions of taxa, observations, media records, references, and geographic features without requiring redesign of the entire platform.

Security

Administrative access, imports, APIs, files, secrets, user submissions, and generated output should be protected through layered controls and auditable workflows.

Long-Term Preservation

Important biological records should remain readable, exportable, attributable, and independently usable beyond the lifetime of any single interface or host.

System Architecture

Layered Core Architecture

Source Layer

The source layer connects to taxonomic catalogues, museum databases, observation platforms, conservation systems, journal repositories, genetic databases, mapping services, institutional archives, and other approved data providers.

Ingestion Layer

The ingestion layer downloads, receives, decodes, parses, validates, fingerprints, and records incoming datasets before they enter the normalized data model.

Normalization Layer

Source-specific structures are mapped into canonical records while preserving the original values and identifiers.

Reconciliation Layer

Names, identifiers, places, taxa, references, media, and records from different sources are compared and linked without erasing disagreement.

Storage Layer

Durable storage maintains normalized entities, original source records, indexes, media metadata, audit history, revisions, relationships, and derived data.

Search Layer

Search indexes support full-text, faceted, geographic, taxonomic, identifier-based, multilingual, and relationship-aware discovery.

Service Layer

APIs and internal services expose records to websites, applications, research tools, maps, exports, and independent clients.

Presentation Layer

Speciedex, SpeciedexExplorer, institutional portals, mobile interfaces, and other clients present the data without owning the underlying scientific model.

Biological Classification

Taxonomy Engine

Taxon Records

A taxon record represents a named biological group or taxonomic concept at a particular rank.

Records may represent domains, kingdoms, phyla, classes, orders, families, genera, species, subspecies, varieties, forms, strains, clades, or other scientifically useful groupings.

Scientific Names

Scientific names are stored separately from taxonomic concepts so that the same name can be interpreted differently across authorities or periods.

Name records can preserve spelling, authorship, publication year, rank, nomenclatural status, source, and formatting rules.

Synonymy

Synonyms, former combinations, misspellings, unavailable names, homonyms, replacement names, and historical classifications can be linked to accepted or source-specific concepts.

Taxonomic Opinions

A source may accept, reject, merge, split, elevate, reduce, or reposition a taxon.

SpeciedexCore records these opinions as attributable assertions rather than rewriting all records into a single unexplained hierarchy.

Hierarchical Relationships

Parent and child relationships allow taxa to be traversed through source-specific or project-selected classifications.

Taxonomic Concepts

Concept records distinguish the biological meaning intended by a name in a specific publication or authority.

This helps separate identical names used with different circumscriptions.

Structured Information

Core Data Model

SpeciedexCore uses linked entities rather than storing all biodiversity information inside a single species table.

Taxa, names, classifications, places, habitats, observations, specimens, media, publications, genetic identifiers, conservation assessments, organizations, people, licenses, sources, and ecological relationships are modeled as distinct records.

These entities can be connected through typed relationships with their own dates, confidence, source, review status, and explanatory notes.

The design allows a single observation, publication, specimen, image, place, or conservation assessment to be linked to more than one relevant record without unnecessary duplication.

Canonical Records

Canonical records provide stable internal entities used by the platform to connect source records, classifications, relationships, and interfaces.

Source Records

Original imported records are preserved separately so that users can trace normalized values back to their source representation.

Assertions

An assertion records a sourced claim such as a habitat association, range statement, conservation status, or taxonomic placement.

Relationships

Typed links represent parentage, synonymy, observation, predation, parasitism, pollination, geography, authorship, citation, and other associations.

Record Identity

Identifier System

Every core record should receive a stable Speciedex identifier independent of its display name.

Scientific names, common names, taxonomic positions, and classifications can change. A stable identifier allows references to remain usable across those changes.

External identifiers from source databases, specimen collections, genetic repositories, journal systems, conservation authorities, mapping systems, and archives are stored as linked identifiers rather than replacing the internal identity of the record.

Identifier mappings may include confidence, source, effective dates, deprecation status, and notes describing uncertain or contested equivalence.

Stable Internal IDs

Internal identifiers remain independent of a taxon's current accepted name or classification.

External Crosswalks

Crosswalk records connect Speciedex entities to identifiers used by independent scientific systems.

Human-Readable Slugs

Web routes may use readable names while redirecting through the stable record identity when names change.

Deprecated IDs

Merged, replaced, or retired identifiers can remain resolvable through redirects and audit history.

Data Quality

Validation Engine

Validation occurs at several stages because a record may be technically well formed while still being scientifically questionable.

Structural validation checks field types, required values, identifiers, encoding, date formats, geometry, relationships, controlled vocabularies, and schema compliance.

Semantic validation checks whether values make sense in context. Examples include impossible coordinates, contradictory ranks, invalid parent relationships, future observation dates, malformed authorship, or conservation records without a defined scope.

Scientific review may still be required for disputed taxonomy, difficult identifications, unusual range records, conflicting literature, or sensitive conservation claims.

Schema Validation

Ensures that records follow the required data structure and field constraints.

Referential Integrity

Ensures that linked records exist and that relationships are valid for their entity types.

Geographic Validation

Checks coordinate ranges, geometry validity, country consistency, elevation, uncertainty, and spatial relationships.

Taxonomic Validation

Checks ranks, parentage, nomenclature, authorities, source identifiers, and classification structure.

Media Validation

Checks file integrity, declared type, dimensions, metadata, license, attribution, and linked records.

Review Workflow

Records can be marked pending, accepted, disputed, rejected, superseded, restricted, or requiring expert review.

Consistent Structure

Normalization Engine

Normalization converts source-specific values into consistent structures without deleting the original source representation.

Dates may be normalized into machine-readable formats, coordinates into standard reference systems, languages into recognized codes, licenses into controlled values, and measurements into consistent units.

Names may be parsed into genus, specific epithet, infraspecific rank, authorship, annotations, and nomenclatural qualifiers.

Normalization rules must be versioned because changes in parsing logic can alter derived output.

Record Matching

Reconciliation Engine

Reconciliation determines whether two source records refer to the same entity, related entities, or different entities with similar labels.

Exact identifiers provide strong evidence, but many sources do not share identifiers.

Matching may therefore consider scientific name, authorship, rank, parent classification, locality, specimen number, publication, source history, geometry, media metadata, and other contextual evidence.

Automated matching can assign confidence scores, but uncertain cases should remain reviewable and reversible.

Exact Matches

Records share a recognized identifier or a complete, verified set of equivalent attributes.

Probable Matches

Records strongly resemble one another but require confirmation before merging or linking.

Related Records

Records may represent a synonym, parent taxon, specimen, observation, former classification, or overlapping concept.

Conflicting Records

Records may disagree in ways that should remain visible rather than being silently collapsed.

Spatial Data

Geography and Mapping Engine

SpeciedexCore stores geographic information as structured records rather than plain text alone.

Geographic entities can include countries, territories, administrative regions, islands, oceans, seas, rivers, watersheds, mountain ranges, caves, protected areas, ecoregions, habitats, localities, and custom research regions.

Observation and specimen records may include coordinates, coordinate uncertainty, elevation, depth, locality text, georeferencing method, precision, date, and source.

Sensitive locations may be obscured, generalized, delayed, or restricted according to conservation and privacy policies.

Point Records

Individual coordinates can represent observations, specimens, nests, recordings, photographs, or sample locations.

Range Geometry

Polygons and multipolygons can represent known, estimated, seasonal, historical, or modeled ranges.

Spatial Relationships

Records can be tested for intersection, containment, distance, overlap, proximity, and regional membership.

Environmental Layers

Climate, elevation, land cover, hydrology, protected areas, and habitat layers may support analysis and visualization.

Evidence of Occurrence

Observation Records

An observation records evidence that an organism or biological event was detected at a particular time and place.

Evidence may include photographs, sound recordings, video, field notes, sensor output, environmental DNA, specimen collection, satellite imagery, tracks, scat, nests, feeding signs, or expert reports.

Observation records can preserve observer identity, source platform, date, time, location, uncertainty, identification, review status, method, equipment, associated media, license, and comments.

Identifications can change without deleting the original observation event.

Identification State

Records may be unidentified, tentatively identified, community identified, expert reviewed, disputed, or rejected.

Evidence Quality

Evidence can be scored or described according to visibility, diagnostic detail, recording quality, metadata completeness, and independent confirmation.

Duplicate Detection

Similar records can be compared using timestamps, location, media hashes, source IDs, and observer information.

Sensitive Observations

Location or observer information may be restricted when disclosure creates conservation, privacy, or safety concerns.

Collection Records

Specimens and Collections

Specimen records connect biological material with institutions, collectors, collection events, preparation methods, catalogue numbers, identifications, media, genetic samples, and publications.

A specimen may be a preserved organism, herbarium sheet, tissue sample, slide, fossil, skeletal element, culture, environmental sample, image-based voucher, or other recognized research object.

Identifications and taxonomic placements may change over time while the underlying specimen record remains stable.

SpeciedexCore can preserve determination history, loan status, institution codes, collection names, storage location, type status, preparation, associated sequences, and digital representations where available.

Digital Assets

Media Management Engine

SpeciedexCore can manage photographs, illustrations, audio, video, maps, diagrams, three-dimensional models, scans, radiographs, specimen images, and other scientific media.

Each asset can preserve its source, creator, copyright, license, attribution text, original URL, checksum, dimensions, duration, format, capture date, location, equipment, linked taxa, and review state.

Derivative files such as thumbnails, previews, normalized audio, transcodes, and web-ready images are linked to the original rather than replacing it.

Media processing should occur in isolated workflows with type verification, resource limits, metadata inspection, and protection against malformed or hostile files.

Original Preservation

Original files can be stored immutably with checksums and source metadata.

Derivatives

Resized images, previews, waveforms, transcodes, and alternate formats can be regenerated from originals.

Attribution

Required creator, source, license, and usage information travels with the asset.

Duplicate Detection

Cryptographic hashes and perceptual comparisons can identify exact and near-duplicate media.

Scientific Literature

References and Evidence

Publications and datasets are modeled as first-class records rather than unstructured citation text alone.

References may include journal articles, monographs, taxonomic descriptions, government reports, museum catalogues, theses, field reports, datasets, conservation assessments, websites, archived pages, and audiovisual sources.

Structured fields can include title, authors, editors, publication date, journal, publisher, volume, issue, pages, language, identifiers, URLs, archive locations, license, and access date.

Assertions throughout SpeciedexCore can link directly to the references that support or dispute them.

Molecular Records

Genetic and Molecular Data

SpeciedexCore may link taxa, specimens, observations, and publications with sequence accessions, barcodes, genomes, genes, markers, assemblies, proteins, and molecular datasets.

Molecular identifiers remain linked to their external repository and source metadata.

Sequence similarity alone does not always determine taxonomic identity. Records may involve contamination, misidentification, hybrids, cryptic species, incomplete lineage sorting, or provisional annotations.

Genetic relationships should therefore be presented with source context, method, specimen linkage, confidence, and scientific interpretation.

Biological Networks

Ecological Relationship Engine

Ecological interactions are stored as attributable, typed relationships between organisms, habitats, places, or environmental conditions.

Relationship types may include predation, herbivory, pollination, seed dispersal, parasitism, infection, mutualism, commensalism, competition, symbiosis, host association, nesting, decomposition, habitat engineering, and food-web connections.

Each relationship can include direction, life stage, geography, season, method, certainty, observation, publication, and explanatory notes.

This structure allows relationships to be queried and visualized without presenting every interaction as universal across all populations or environments.

Change History

Versioning and Audit History

Scientific records change as new sources are published, classifications are revised, errors are corrected, and observations are reviewed.

SpeciedexCore is designed to preserve revision history rather than overwriting important changes without explanation.

Each revision can record the changed fields, previous values, new values, actor, timestamp, source, reason, review state, software version, and associated import job.

Releases may be assigned version identifiers so that datasets, analyses, publications, and exports can cite the exact state of the platform used.

Record History

Tracks changes to individual taxa, observations, places, media, references, and relationships.

Import History

Records which source release, adapter version, and transformation rules created or changed a record.

Release Snapshots

Produces reproducible snapshots for archival, scientific, and deployment use.

Reversion

Authorized operators can restore prior states when an import, edit, or automated process produces incorrect results.

Access Control

Authentication and Authorization

Public biodiversity data may be readable without an account, while contribution, moderation, import, administration, and sensitive-data access require authenticated permissions.

Access should follow least-privilege principles. A user or service receives only the permissions required for its role.

Administrative credentials, signing keys, API secrets, database credentials, backup keys, and storage tokens must remain outside public repositories and web-accessible directories.

Sensitive actions should be logged and may require stronger authentication, secondary approval, or signed release workflows.

Distributed Infrastructure

Synchronization and Mirrors

SpeciedexCore is intended to support independent mirrors, institutional deployments, offline archives, regional nodes, and specialized research copies.

Synchronization can operate through signed releases, incremental change feeds, content-addressed files, database replication, export packages, or documented import workflows.

Mirrors may replicate the entire public dataset or only selected taxonomic, geographic, media, or institutional subsets.

Sensitive records, private submissions, credentials, and restricted conservation data should not be included in public replication streams.

Full Mirrors

Disconnected Access

Offline and Field Operation

Biodiversity work often occurs in locations with limited, expensive, intermittent, or unavailable connectivity.

SpeciedexCore can support offline datasets, local search indexes, cached maps, field forms, media collection, and delayed synchronization.

Field devices may create observations, photographs, recordings, notes, specimens, and geographic records locally before transferring them to a connected system.

Synchronization workflows should preserve the original field identifiers, timestamps, device information, evidence, and edit history while resolving conflicts safely.

Resilience

Backup and Recovery

SpeciedexCore separates production data, original source files, media originals, indexes, configuration, secrets, logs, and generated derivatives so that each can be protected appropriately.

Backups should include versioned database snapshots, source archives, media manifests, checksums, encryption, off-site copies, restore documentation, and periodic recovery tests.

Search indexes and media derivatives may be rebuilt from authoritative data, while source files and original media require stronger preservation.

A backup is not considered reliable until restoration has been tested.

Database Backups

Preserve structured records, revisions, relationships, permissions, and system configuration.

Source Archives

Preserve the exact source files used to create each imported release.

Media Originals

Preserve original media independently from generated previews and transcodes.

Recovery Drills

Verify that documented procedures can restore a usable system on independent infrastructure.

Scale and Reliability

Performance Architecture

SpeciedexCore is designed to separate durable data storage from search indexes, caches, media delivery, background jobs, and public request handling.

Expensive import, reconciliation, image processing, geospatial analysis, and index-building tasks can run as controlled background jobs rather than blocking public requests.

Frequently accessed public records may be cached while preserving clear invalidation rules and version identifiers.

Performance improvements should not bypass access controls, scientific provenance, record versioning, or source traceability.

Pagination

Large result sets are paginated or streamed rather than loaded into memory at once.

Background Jobs

Imports, exports, media processing, and indexing run through controlled queues.

Caching

Public records and derived views can be cached with explicit version and expiration policies.

Incremental Indexing

Only changed records need to be reindexed after ordinary updates.

Defensive Engineering

Security Architecture

SpeciedexCore processes data, files, media, and user submissions from many sources. Every external input must be treated as untrusted until validated.

Public services should be separated from administrative systems, import workers, databases, signing keys, backup storage, and sensitive conservation records.

File processing should use allowlisted formats, type verification, size limits, timeouts, resource controls, isolated workers, and secure temporary storage.

Logs should support incident investigation without exposing credentials, private data, restricted coordinates, or unnecessary personal information.

Least Privilege

Services and users receive only the permissions required for their assigned tasks.

Network Segmentation

Public, administrative, database, storage, and backup systems can be isolated from one another.

Input Validation

API input, uploads, URLs, archives, markup, queries, and metadata are validated before use.

Secret Management

Credentials, signing keys, tokens, and encryption keys remain outside source code and public files.

Audit Logging

Privileged actions, imports, releases, permission changes, and data restrictions are recorded.

Dependency Review

Libraries, runtimes, parsers, codecs, and plugins are maintained, pinned, reviewed, and updated.

Responsible Data Use

Privacy and Sensitive Data

SpeciedexCore may process observer names, contributor accounts, institutional contacts, private localities, unpublished research, restricted specimen data, and sensitive conservation records.

The platform should collect only the personal information required for its legitimate scientific, operational, and security purposes.

Public exports should omit private credentials, internal notes, restricted coordinates, unpublished submissions, security metadata, and personal information not approved for release.

Access to sensitive records should be attributable, reviewable, limited by role, and revocable.

Extensibility

Adapter and Plugin System

SpeciedexCore can support modular adapters for data sources, storage systems, search backends, mapping tools, authentication providers, media processors, exporters, and scientific workflows.

Plugins should use documented interfaces and receive only the permissions and data required for their function.

Third-party extensions must not be treated as trusted merely because they are compatible with the platform.

Installations should review plugin source, dependencies, maintenance history, licenses, network access, file access, data access, and update practices before use.

Assisted Analysis

AI and Machine-Learning Integration

SpeciedexCore may use machine learning to assist with image classification, acoustic recognition, literature extraction, metadata normalization, duplicate detection, taxonomic reconciliation, translation, anomaly detection, and quality review.

Machine-generated results are stored as attributed model output with version, confidence, input, date, and review status.

AI output should not silently replace expert identifications, source data, conservation assessments, or taxonomic decisions.

Models can inherit errors from mislabeled training data, geographic bias, incomplete taxonomic coverage, poor media quality, or outdated classifications.

Interoperability

Open Standards and Data Formats

SpeciedexCore is intended to use established scientific and technical standards where they accurately represent the required data.

Supported structures may include Darwin Core, controlled vocabularies, recognized geographic formats, standard date and language codes, common publication identifiers, documented JSON schemas, delimited exports, and interoperable media metadata.

No single standard represents every scientific concept required by SpeciedexCore.

Extensions should remain documented, versioned, exportable, and mapped to existing standards where practical.

Rights and Attribution

Licensing and Source Terms

SpeciedexCore software, project documentation, imported datasets, media, scientific publications, source records, and generated exports may each be governed by different licenses and usage terms.

The presence of a record inside SpeciedexCore does not automatically place the underlying source material into the public domain.

Exports should preserve required attribution, source identifiers, license notices, and restrictions.

Restricted, non-redistributable, confidential, embargoed, or private records should not be included in public releases without authorization.

Public Infrastructure

Open-Source Philosophy

Biodiversity knowledge should not depend entirely upon a proprietary interface, closed vendor, inaccessible file format, or single institution.

Open-source software allows researchers, institutions, developers, educators, and independent operators to inspect, adapt, preserve, and improve the technical infrastructure.

Public code does not guarantee secure or scientifically accurate operation by itself. Review, testing, documentation, governance, maintenance, and responsible deployment remain necessary.

SpeciedexCore is intended to combine open software with source attribution, scientific transparency, operational discipline, and long-term preservation.

Scientific and Technical Notice

SpeciedexCore is an evolving research and software platform.

Biological classification, conservation status, geographic range, scientific interpretation, source availability, identifiers, and technical interfaces may change.

Records may be incomplete, provisional, disputed, machine-assisted, outdated, source-limited, or awaiting scientific review.

Software defects, import errors, source changes, configuration mistakes, dependency failures, incomplete backups, or incorrect permissions may affect operation.

Users should verify important scientific information through current primary literature, recognized authorities, source databases, qualified specialists, and appropriate institutional procedures.

SpeciedexCore should not be used as the sole basis for medical, veterinary, legal, regulatory, quarantine, wildlife enforcement, trade, public-safety, or conservation decisions.
