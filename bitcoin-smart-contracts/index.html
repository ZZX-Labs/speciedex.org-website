Programmable Bitcoin

Bitcoin Smart Contracts

Bitcoin smart contracts are programmable spending conditions enforced by the Bitcoin network.

Rather than executing arbitrary application code on every node, Bitcoin uses a deliberately constrained scripting system to define when, how, and by whom a transaction output may be spent.

These mechanisms support multisignature custody, inheritance, escrow, payment channels, timelocked recovery, atomic swaps, vaults, Lightning Network payments, Discreet Log Contracts, and other financial systems built directly upon Bitcoin.

What a Bitcoin Smart Contract Is

A Bitcoin smart contract is a set of cryptographic and logical conditions attached to bitcoin.

These conditions determine what evidence must be provided before the funds can be spent. That evidence may include a valid digital signature, multiple signatures, the passage of time, knowledge of a secret, or satisfaction of a more complex spending policy.

Once a transaction is broadcast, full nodes evaluate the relevant conditions. Transactions that fail those conditions are rejected.

Bitcoin Script

Bitcoin smart contracts are primarily expressed using Bitcoin Script, a stack-based scripting language designed for transaction validation.

Script is intentionally limited. It is not designed as a general-purpose programming language and does not include unrestricted loops or arbitrary persistent computation.

This conservative design reduces complexity, limits resource consumption, improves predictability, and helps make validation consistent across the network.

Design Philosophy

Why Bitcoin Uses a Constrained Language

Every fully validating Bitcoin node must be able to evaluate transaction conditions consistently.

A highly expressive execution environment can introduce unpredictable computation, denial-of-service risk, software ambiguity, complex state management, and a larger attack surface.

Bitcoin instead favors narrowly defined conditions whose meaning can be reviewed and verified before funds are committed.

The result is a smart-contract model focused less on running large applications inside the blockchain and more on securely coordinating ownership, authorization, time, secrets, payments, and settlement.

Contract Components

Core Smart-Contract Building Blocks

Digital Signatures

Signatures prove that a transaction was authorized by the holder of a corresponding private key.

Most Bitcoin spending conditions require one or more valid signatures.

Multisignature

Multisignature policies require more than one key to authorize a transaction.

Common forms include two-of-three, three-of-five, and other threshold arrangements.

Timelocks

Timelocks prevent funds from being spent until a specified block height, time, or relative delay has passed.

Hashlocks

Hashlocks require a spender to reveal a secret whose hash matches a value committed to in advance.

Threshold Policies

Threshold policies allow one of several approved combinations of conditions to authorize spending.

Recovery Paths

Contracts can include alternate spending paths that become available after a delay or under emergency conditions.

Commitments

Cryptographic commitments allow parties to prove that specific data or terms existed without revealing all information immediately.

Transaction Chains

Multiple transactions can be constructed together so that later actions depend upon earlier signed states.

Shared Authorization

Multisignature Contracts

Shared Custody

A multisignature contract can require approval from several independent keys before bitcoin may be spent.

A two-of-three arrangement, for example, permits any two authorized keys to approve a transaction while preventing a single compromised key from moving the funds.

Organizational Treasury

Businesses, research organizations, nonprofits, and collaborative projects may use threshold signatures to distribute financial authority across multiple people or devices.

This can reduce dependence upon a single employee, administrator, machine, office, or physical location.

Personal Security

Individuals may store separate keys on different hardware wallets, in different locations, or with carefully chosen recovery participants.

A properly designed threshold policy can make theft more difficult while preserving recovery options.

Collaborative Custody

Collaborative custody can divide authority between a user and one or more service providers without giving any single provider unilateral control.

The exact security model depends upon the contract, wallet software, backup procedures, and distribution of keys.

Time-Based Conditions

Timelocks

Absolute Timelocks

Absolute timelocks prevent a transaction or spending path from becoming valid before a specified block height or time.

They can be used for delayed distributions, scheduled recovery, long-term savings, inheritance planning, or contract deadlines.

Relative Timelocks

Relative timelocks require a specified delay after a previous transaction output has been confirmed.

These delays are important in payment channels, vault designs, revocation systems, and staged recovery arrangements.

Delayed Recovery

A contract may allow an ordinary spending path with one key while permitting a separate recovery key to act only after a delay.

This can provide time to detect unauthorized activity and respond before final settlement.

Inheritance

Timelocked recovery paths can support inheritance arrangements in which designated keys gain authority only after a long period of inactivity or according to an established succession plan.

Any inheritance design should be reviewed carefully for legal, operational, privacy, and security risks.

Secret-Based Conditions

Hashlocks

A hashlock requires the spender to reveal a secret value, commonly called a preimage, whose cryptographic hash matches a value specified by the contract.

Before the secret is revealed, parties can verify the committed hash without knowing the underlying value.

When the correct preimage is published in a valid transaction, the secret becomes visible and may unlock a related contract elsewhere.

Hashlocks are important in Lightning Network payments, atomic swaps, cross-system settlement, and contracts in which one disclosure triggers several coordinated outcomes.

Hash and Time

Hashed Timelock Contracts

A Hashed Timelock Contract combines a hashlock with a timelock.

One party may claim the funds by revealing the correct secret before a deadline. If that does not happen, an alternate party may recover the funds after the timeout.

This creates a conditional payment that either succeeds through disclosure of the required preimage or safely expires through a refund path.

Hashed Timelock Contracts historically formed a major component of routed Lightning Network payments and remain a foundational concept in Bitcoin contract design.

Payment Channels

Lightning Network Smart Contracts

Channel Funding

A Lightning channel begins when bitcoin is committed to an on-chain funding output governed by a shared contract.

The participants then exchange signed state updates representing changes in channel ownership without publishing every update to the blockchain.

Commitment Transactions

Each participant holds a transaction capable of settling the current channel state on-chain.

Contract conditions and revocation mechanisms discourage publication of obsolete states.

Routed Payments

Conditional payments can pass through several Lightning nodes.

Each routing participant forwards value only when the linked conditions are satisfied, allowing the payment to complete atomically across the route.

Cooperative Settlement

Participants may cooperatively close a channel by agreeing upon the final allocation of funds and publishing a settlement transaction.

Unilateral recovery paths remain available when a counterparty is unavailable or uncooperative.

Conditional Settlement

Escrow and Dispute Resolution

Bitcoin contracts can support escrow arrangements in which funds require authorization from a buyer, seller, mediator, or threshold combination of participants.

A common two-of-three arrangement gives the buyer and seller the ability to complete a transaction cooperatively while allowing a neutral third party to assist if a dispute occurs.

Unlike ordinary custodial escrow, the mediator does not necessarily need unilateral control over the funds.

Contract design cannot determine whether physical goods were delivered, whether work met a subjective standard, or whether a legal obligation was fulfilled. Human judgment, documentation, trusted evidence, or external dispute-resolution processes may still be required.

Coordinated Exchange

Atomic Swaps

Atomic swaps allow two parties to exchange assets across compatible systems without requiring one party to send irrevocably first.

Hashlocks and timelocks can coordinate the exchange so that either both sides complete or both sides eventually recover their original funds.

When one party claims the asset on one system, the revealed secret permits the other party to claim the corresponding asset on the second system.

Atomic swaps require compatible cryptographic primitives, carefully coordinated timeouts, reliable transaction monitoring, sufficient fees, and correct implementation.

Modern Bitcoin Contracts

Taproot

Key-Path Spending

Taproot allows participants to aggregate cooperative authorization into a spending path that resembles a single public key and signature.

When all required parties agree, complex arrangements may settle efficiently without revealing every possible contract branch.

Script-Path Spending

Taproot can commit to multiple alternative scripts while revealing only the branch actually used.

Unused conditions remain undisclosed, improving privacy and reducing the amount of contract data placed on-chain.

Schnorr Signatures

Taproot uses Schnorr signatures, which support efficient signature aggregation and cleaner construction of collaborative signing schemes.

Improved Contract Design

Taproot enables more private and efficient multisignature policies, payment channels, vault structures, recovery paths, and advanced signing arrangements.

Structured Policy Design

Miniscript

Miniscript is a structured language and framework for expressing Bitcoin spending policies in a form that can be analyzed, composed, and compiled into Bitcoin Script.

It helps wallet developers reason about which keys, timelocks, and threshold conditions can satisfy a contract.

Miniscript can also assist with fee estimation, satisfiability analysis, recovery planning, wallet interoperability, and construction of complex policies without manually assembling low-level script operations.

A policy may describe a primary spending path, one or more recovery paths, delayed emergency authority, and alternative combinations of keys.

Wallet Interoperability

Output Script Descriptors

Output script descriptors provide a standardized way to describe how wallet addresses and spending scripts are derived.

A descriptor can identify the script type, public keys, derivation paths, multisignature threshold, checksum, and other information needed to reconstruct a wallet policy.

Descriptors improve backup quality and interoperability because they preserve more information than a collection of addresses alone.

A watch-only system may use public descriptors to monitor balances and generate addresses without holding the private keys required to spend.

External Events

Discreet Log Contracts

Discreet Log Contracts allow parties to create Bitcoin contracts whose settlement depends upon an event attested to by an external oracle.

The oracle publishes cryptographic information associated with an outcome, such as a price, measurement, election result, weather observation, or other verifiable event.

The oracle does not necessarily take custody of funds and may not know the identities or exact contract terms of the participants.

Discreet Log Contracts can support conditional payouts, financial agreements, insurance-like arrangements, hedging, research incentives, and prediction-based settlement.

Participants remain exposed to oracle failure, misreporting, collusion, ambiguous outcomes, software defects, regulatory obligations, and counterparty risk.

External Information

Oracles and Real-World Data

Bitcoin nodes can verify signatures, transaction history, block conditions, hashes, and other information available within the protocol.

They cannot independently determine whether a shipment arrived, whether rain fell in a specific location, whether a species was observed, whether a sporting event occurred, or whether a published market price is correct.

An oracle provides a cryptographically signed statement about an external event.

Oracle-based systems must consider source reliability, measurement standards, publication timing, data availability, conflicting reports, compromise, jurisdiction, and dispute resolution.

Multiple independent oracles and threshold attestation can reduce dependence upon a single data provider, but they do not eliminate the need for trust in external information.

Defensive Custody

Bitcoin Vaults

A Bitcoin vault is a custody design intended to restrict or delay how funds can be moved.

Vault systems may separate ordinary spending authority from emergency recovery authority, require waiting periods, limit destination addresses, or provide time to detect and respond to an unauthorized transaction.

Some vault designs can be approximated using existing multisignature, timelock, pre-signed transaction, and wallet-policy techniques.

More advanced vault proposals may depend upon future consensus changes or additional transaction constraints.

Vaults can improve security against certain threats while introducing additional complexity, backup requirements, monitoring obligations, fee considerations, and recovery failure modes.

Long-Term Recovery

Inheritance and Succession Contracts

Bitcoin inheritance requires both technical and legal planning.

A smart-contract policy may include delayed recovery keys, distributed family authorization, professional executors, multisignature trustees, or prearranged succession paths.

The contract must be designed so that heirs can recover the funds when appropriate without gaining premature access or exposing sensitive information while the owner is alive.

Plans should account for lost devices, forgotten passphrases, obsolete wallet software, changing family circumstances, unavailable cosigners, legal disputes, and long periods between testing.

Any inheritance arrangement should be documented, periodically tested, and reviewed with qualified legal and financial professionals where appropriate.

Scientific and Public Infrastructure

Research and Conservation Applications

Research Grants

Multisignature and milestone-based contracts may help coordinate research grants, field-expedition funding, equipment procurement, and staged project support.

Conservation Bounties

Conditional payouts may support verified habitat restoration, species observations, invasive-species removal, cleanup programs, or independently audited conservation outcomes.

Data Publication

Cryptographic commitments and timestamped transactions may help establish that a dataset, research record, image, or document existed before a particular point in time.

Distributed Governance

Threshold authorization can help distribute control of shared project funds across institutions, researchers, trustees, or regional conservation partners.

Emergency Funds

Timelocked and multisignature arrangements may help secure emergency response funds while preserving access during disasters, infrastructure failures, or organizational disruption.

Provenance

Signed records can help establish the origin, authorship, approval history, and integrity of scientific or conservation documentation.

Speciedex may research Bitcoin smart contracts for transparent project funding, multisignature treasury management, Lightning payments, public bounties, contributor compensation, archival timestamping, data provenance, infrastructure procurement, and conservation financing.

Limitations

What Smart Contracts Cannot Determine

A smart contract can enforce only the conditions visible to its validation environment.

It cannot independently determine whether a biological specimen was correctly identified, whether a conservation project succeeded, whether a shipment contained the promised equipment, whether a person acted ethically, or whether a legal agreement was fair.

External facts require trusted data, human judgment, sensors, signed attestations, legal processes, oracles, inspectors, auditors, or scientific review.

Smart contracts reduce certain forms of counterparty risk, but they do not eliminate fraud, coercion, bad data, operational mistakes, legal disputes, software defects, or physical-world uncertainty.

Security

Smart-Contract Risks

Incorrect Policy

A contract may faithfully enforce conditions that do not match the user's actual intent.

Lost Keys

If required keys are lost and no recovery path exists, the funds may become permanently unspendable.

Compromised Keys

Attackers who obtain enough authorized keys may be able to satisfy the contract and transfer the funds.

Backup Failure

Incomplete descriptors, missing derivation details, lost channel state, or damaged recovery records may prevent restoration.

Software Defects

Wallets, signing devices, contract compilers, node software, and interfaces may contain implementation errors.

Fee Conditions

A contract may become expensive or difficult to execute if transaction fees rise unexpectedly.

Privacy Leakage

Contract structures, public keys, timing, transaction relationships, and revealed scripts may expose information about participants.

Oracle Failure

Contracts based upon external events may fail if an oracle disappears, reports incorrectly, or becomes compromised.

Test Before Funding

Complex Bitcoin contracts should be reviewed, tested on appropriate development networks, independently verified, backed up, and exercised through recovery procedures before substantial funds are committed.

Responsible Engineering

Contract Design Principles

Use the simplest policy that satisfies the actual security requirement.

Prefer widely reviewed standards, mature wallet software, documented descriptors, reproducible builds, and independently verifiable open-source implementations.

Separate signing devices, distribute critical keys, document recovery procedures, protect metadata, verify receiving addresses, and test every recovery path.

Avoid relying upon a single developer, service provider, hardware device, oracle, geographic location, or undocumented custom script.

Preserve clear human-readable documentation explaining who can spend, under what conditions, after what delay, and how recovery is performed.

Development

Bitcoin Contract Development

Regtest

Bitcoin Core's regression-test mode provides a private local blockchain where developers can create blocks, test wallets, construct transactions, and inspect contract behavior without using real bitcoin.

Signet

Signet provides a public testing environment with more predictable block production than traditional proof-of-work test networks.

PSBT

Partially Signed Bitcoin Transactions allow multiple wallets and devices to exchange transaction information and signatures without exposing private keys.

Hardware Wallets

Hardware signing devices can isolate private keys and display transaction details before authorization.

Watch-Only Wallets

Watch-only systems can monitor contract balances and prepare transactions without holding spending keys.

Independent Verification

Full nodes, separate wallet software, policy analysis, transaction decoders, and reproducible test vectors can help verify contract behavior.
