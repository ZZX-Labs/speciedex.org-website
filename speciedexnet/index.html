Distributed Biodiversity Infrastructure

SpeciedexNet

SpeciedexNet is the distributed synchronization, mirroring, archival, regional-delivery, and data-exchange network for the Speciedex biodiversity ecosystem.

It is designed to distribute public biodiversity records, taxonomic indexes, scientific references, geographic data, media metadata, search indexes, documentation, software releases, and verified data snapshots across independently operated nodes.

SpeciedexNet supports resilient access, institutional mirrors, regional services, offline archives, field deployments, content verification, independent research systems, and long-term preservation.

What SpeciedexNet Is

SpeciedexNet is the network and distribution layer of the Speciedex project.

It allows biodiversity data and related services to be replicated across multiple locations rather than depending upon a single host, organization, domain, or data center.

Nodes may maintain complete public mirrors, regional subsets, institutional collections, archival snapshots, media caches, search indexes, API gateways, or specialized scientific datasets.

Why a Distributed Network Is Needed

Scientific infrastructure can be disrupted by outages, funding loss, policy changes, hardware failure, domain loss, institutional closure, censorship, natural disaster, conflict, or unauthorized modification.

Replication across independent operators reduces the risk that valuable biodiversity information will become inaccessible or disappear entirely.

Distribution also improves regional performance, supports low-connectivity environments, and allows institutions to preserve locally relevant data.

Network Mission

Preserve and Distribute Biodiversity Knowledge

SpeciedexNet exists to make biodiversity information more resilient, verifiable, portable, and geographically accessible.

The network is designed to preserve public data, documentation, software, indexes, source manifests, release history, checksums, and supporting metadata in forms that can be independently stored and validated.

A functioning Speciedex installation should be able to recover from verified releases and documented source archives without relying upon one central server.

The project also seeks to provide institutions, researchers, educators, conservation organizations, and regional operators with practical tools for hosting trusted mirrors and local services.

Engineering Principles

SpeciedexNet Design Principles

Independent Operation

Nodes should remain capable of serving verified public data without continuous dependence upon the primary installation.

Verifiable Content

Releases, files, manifests, indexes, and archives should be independently checkable through hashes, signatures, and documented metadata.

Source Transparency

Replicated records should preserve source identifiers, attribution, licenses, retrieval dates, and release history.

Least Privilege

Replication clients, mirrors, gateways, and operators should receive only the permissions required for their function.

Graceful Degradation

Nodes should continue serving useful static records, documentation, and cached search data when advanced services are unavailable.

Regional Adaptability

Operators may maintain selected geographic, linguistic, taxonomic, media, or institutional subsets.

Open Interoperability

Synchronization and export formats should be documented and usable by independent software.

Long-Term Preservation

Important releases should remain readable and recoverable beyond the lifetime of a particular application or provider.

Data Exchange

Synchronization

SpeciedexNet synchronization transfers approved public records and related files between publishers, mirrors, archives, and regional nodes.

A node may synchronize full releases, incremental updates, selected taxonomic groups, geographic subsets, source archives, software packages, indexes, documentation, or media manifests.

Synchronization should be resumable, version-aware, independently verifiable, and safe to repeat.

Failed or incomplete transfers must not replace a valid local release until integrity checks have passed.

Full Synchronization

Downloads a complete verified release and associated public metadata.

Incremental Synchronization

Transfers only records and objects changed since a known release or checkpoint.

Selective Synchronization

Replicates selected regions, taxa, languages, media classes, sources, or record types.

Offline Transfer

Moves signed release packages through removable media or controlled physical transport.

Versioned Distribution

Release and Snapshot Model

Public SpeciedexNet data is distributed as identifiable releases and snapshots.

Each release can include a version identifier, publication timestamp, schema version, source manifest, record counts, included components, checksums, signatures, migration notes, and compatibility information.

Nodes should be able to retain multiple releases so that researchers can reproduce earlier analyses or recover after a faulty update.

Release metadata should distinguish authoritative project releases from community mirrors, experimental datasets, institutional additions, and locally modified copies.

Stable Releases

Reviewed snapshots intended for public use, replication, citation, and long-term preservation.

Candidate Releases

Pre-release snapshots used for validation, testing, and operator review.

Historical Snapshots

Preserved versions supporting scientific reproducibility and audit history.

Regional Packages

Versioned subsets focused on selected geography, taxonomy, language, or institutional needs.

Release Metadata

Manifests and Inventory

Every distributed release should include a machine-readable manifest describing its contents.

A manifest may identify files, object hashes, sizes, formats, compression, schemas, record counts, source versions, licenses, signatures, dependencies, and required software versions.

Nodes use the manifest to determine which objects are missing, changed, corrupted, unsupported, or no longer part of the current release.

Manifests also support selective synchronization by allowing operators to request only approved categories or subsets.

Content Verification

Integrity Checking

SpeciedexNet uses cryptographic hashes and signed metadata to detect corruption, incomplete transfer, or unauthorized modification.

A node should verify downloaded files before publishing them or importing them into active services.

Hashes verify content equality, while digital signatures can verify that a release manifest was approved by a recognized publisher.

Verification does not by itself prove scientific accuracy. It proves that the received content matches the signed or referenced release.

File Hashes

Verify individual files, archives, indexes, and media objects.

Manifest Hashes

Verify that the release inventory has not changed.

Digital Signatures

Verify release approval against trusted public keys.

Audit Logs

Record download, verification, activation, rollback, and publication events.

Immutable Objects

Content-Addressed Storage

SpeciedexNet may identify immutable objects by the cryptographic hash of their content.

Content addressing allows nodes to recognize duplicate files, retrieve objects from multiple mirrors, verify transfers, and preserve identical objects under a stable identity.

Human-readable names and URLs can point to content-addressed objects while release manifests define which object belongs to a particular version.

Mutable records remain versioned through SpeciedexCore and should not be confused with immutable release objects.

Taxonomic Distribution

Taxonomy Synchronization

Taxonomic records require version-aware synchronization because names, ranks, synonymy, parentage, and accepted classifications can change.

SpeciedexNet distributes taxonomic concepts, names, relationships, authorities, source opinions, identifiers, and revision metadata as linked records.

Nodes should not overwrite locally selected classifications without preserving source and version context.

Institutional nodes may add local annotations or alternate classifications while maintaining a clear distinction between project releases and local extensions.

Distributed Discovery

Search Federation

SpeciedexNet may federate search across independent nodes without requiring every node to host every record.

A regional search service may answer local queries from its own index and forward broader requests to trusted public gateways.

Federated results should identify the responding node, dataset version, record source, query scope, and any differences in ranking or coverage.

Query federation must use limits, timeouts, validation, privacy controls, and protection against recursive or abusive requests.

Local Search

Searches the dataset and indexes maintained by the current node.

Trusted Federation

Queries approved remote nodes using documented protocols and bounded request policies.

Result Aggregation

Combines results while retaining node, source, version, and confidence metadata.

Query Caching

Caches permitted public results to improve performance and reduce repeated remote requests.

Service Delivery

API Gateways

API gateways provide controlled public and institutional access to Speciedex services.

Gateways may route requests to local services, regional nodes, read replicas, search nodes, object storage, or static exports.

They can enforce authentication, authorization, rate limits, request size, timeouts, version compatibility, logging, and response caching.

Administrative services and sensitive data should never be exposed merely because a public gateway can reach the same internal network.

Regional Delivery

Geographic Distribution

SpeciedexNet can place public data and services closer to the users, institutions, and environments that depend upon them.

Regional nodes can improve latency, reduce international bandwidth use, support local languages, and preserve locally important biodiversity records.

Geographic routing may consider node health, proximity, available datasets, language, legal restrictions, and service capability.

Users should remain able to identify which node served a record and which release it used.

Efficient Delivery

Caching Strategy

Public records, generated pages, static assets, map tiles, search responses, thumbnails, software releases, and documentation may be cached.

Cache entries should include clear version identifiers, expiration rules, invalidation behavior, access scope, and source metadata.

Private, restricted, personalized, administrative, or sensitive responses must not enter shared public caches.

Immutable release objects can be cached for long periods, while mutable API responses require shorter and explicitly controlled lifetimes.

Scientific Media

Media Distribution

Media nodes can distribute licensed photographs, audio, video, maps, scans, illustrations, and three-dimensional models.

Original files, normalized derivatives, thumbnails, previews, waveforms, and transcodes should remain linked through verified manifests.

Every distributed item should preserve creator, source, license, attribution, checksum, linked record, and technical metadata.

Mirrors must not redistribute media beyond the rights granted by the original license or source agreement.

Low-Connectivity Access

Offline Distribution

SpeciedexNet supports environments where continuous internet access is unavailable, unreliable, expensive, or inappropriate.

Offline packages may contain regional species records, taxonomy, maps, field references, conservation data, educational materials, search indexes, software, and permitted media.

Packages can be distributed through removable storage, local network appliances, research-station servers, classroom systems, or controlled field devices.

Each package should remain independently verifiable and identify its release date, geographic scope, taxonomy, licenses, and update path.

Institutional Data

Local Extensions

Institutional nodes may maintain local records that are not part of the primary public Speciedex release.

Examples include unpublished observations, restricted collection data, local annotations, institutional identifiers, draft research, private media, and region-specific classifications.

Local records must remain clearly marked so users can distinguish them from signed Speciedex releases.

Institutions may submit approved records upstream through documented contribution and review workflows.

Service Resilience

High Availability and Fault Tolerance

SpeciedexNet is designed so that the failure of one service or node does not make the entire platform unavailable.

Static mirrors can continue serving public pages and data exports while dynamic services recover.

Search gateways can route requests to healthy indexes, and object requests can be retrieved from alternate verified mirrors.

Health checks, bounded retries, circuit breakers, timeouts, failover policies, and degraded-service modes should prevent cascading failures.

Service Redundancy

Multiple nodes can provide equivalent public services.

Read Replicas

Public queries can be served from independent read-only copies.

Static Fallback

Essential records and documentation remain available when dynamic services fail.

Controlled Failover

Traffic moves only to nodes with compatible releases and verified health.

Recovery Infrastructure

Backup and Recovery

Replication improves availability, but replication alone is not a complete backup strategy.

A faulty deletion, corrupted release, unauthorized change, or broken synchronization process can be copied across multiple nodes.

SpeciedexNet operators should maintain versioned, encrypted, off-site, and independently restorable backups.

Recovery procedures should be tested on clean infrastructure and documented for databases, source archives, media originals, manifests, keys, configuration, and software.

Verification and Authority

Trust Model

SpeciedexNet does not assume that every reachable node is trusted.

Nodes may be official publishers, approved mirrors, institutional extensions, independent archives, experimental deployments, or unknown third parties.

Clients should evaluate release signatures, node identity, dataset version, compatibility, source provenance, and operator policy before accepting data.

Signed releases establish origin and integrity but do not remove the need for scientific review, source evaluation, or local policy.

Trusted Publishers

Authorized keys sign official project releases and manifests.

Approved Mirrors

Known operators replicate official content and follow documented compatibility policies.

Institutional Extensions

Institutions sign or identify locally maintained records separately from project releases.

Untrusted Sources

Unknown nodes may be queried only through restricted, validated, and clearly labeled workflows.

Node Identity

Authentication and Authorization

Public mirrors may serve unrestricted data anonymously, while publishing, synchronization administration, private federation, restricted-data exchange, and node management require authenticated access.

Node identity may be established through keys, certificates, signed requests, mutually authenticated channels, or other documented methods.

Permissions should be separated among publishing, replication, reading, monitoring, administration, signing, and recovery operations.

Signing keys, backup keys, administrative credentials, API secrets, and storage tokens must remain outside public repositories and web-accessible paths.

Responsible Replication

Privacy and Sensitive Data

Public synchronization streams must exclude private accounts, credentials, unpublished submissions, restricted coordinates, administrative notes, internal security metadata, and personal information not approved for release.

Sensitive biodiversity data may require generalized geography, delayed publication, limited node access, or exclusion from public mirrors.

Restricted replication should use explicit access controls, purpose limitation, audit logs, retention policies, and revocable authorization.

Node operators remain responsible for applicable privacy, conservation, institutional, licensing, and data-use obligations.

Defensive Network Engineering

Security Architecture

SpeciedexNet receives files, manifests, requests, synchronization messages, media, and metadata from remote systems. All external input must be treated as untrusted.

Synchronization should use validated schemas, allowlisted formats, strict size limits, bounded decompression, timeouts, secure temporary storage, checksum verification, and isolated processing.

Public gateways, administrative systems, signing systems, databases, backup storage, and restricted records should be separated through network and permission boundaries.

Operators should maintain dependency updates, patching, logging, alerting, access review, key rotation, and incident-response procedures.

Input Validation

Validate requests, URLs, manifests, archives, file names, metadata, and API parameters.

Resource Limits

Limit archive expansion, memory, storage, CPU, query cost, and request duration.

Network Segmentation

Separate public delivery, synchronization, storage, administration, signing, and backups.

Key Protection

Store signing and administrative keys outside public application servers where practical.

Audit Logging

Record synchronization, activation, rollback, permission, and signing operations.

Rate Limiting

Protect gateways and federation interfaces from abuse and accidental overload.

Release Verification

Reject incomplete, unsigned, incompatible, or corrupted releases according to node policy.

Incident Recovery

Support key revocation, node isolation, rollback, re-verification, and clean restoration.

Service Protection

Rate Limiting and Fair Use

Public APIs, search gateways, media services, exports, and synchronization endpoints may enforce rate limits.

Limits can protect services from accidental overload, aggressive crawlers, abusive extraction, expensive queries, repeated large exports, or denial-of-service activity.

Limits may differ for anonymous users, authenticated researchers, approved mirrors, institutional partners, and internal services.

Bulk public data should be obtained through release archives and documented exports rather than repeatedly reconstructing the complete dataset through individual API requests.

Network Operations

Monitoring and Node Health

SpeciedexNet operators should monitor availability, release version, synchronization lag, storage capacity, verification failures, query latency, error rates, certificate status, and service dependencies.

Health information should distinguish between complete failure, degraded service, stale data, unsupported release, and temporary maintenance.

Public status data should not expose credentials, internal addresses, restricted systems, or details that materially weaken security.

Monitoring records can support capacity planning, failover, incident response, and identification of unreliable mirrors.

Durable Service Design

Static and Dynamic Hybrid Architecture

SpeciedexNet can combine durable static content with dynamic services.

Static pages, software releases, documentation, manifests, and public data packages can be mirrored easily and served with minimal infrastructure.

Dynamic services provide search, APIs, maps, account features, contributions, synchronization management, and current data views.

When dynamic systems are unavailable, static mirrors can continue providing essential public information and recovery materials.

Static Distribution

Static Hosting Integration

Selected SpeciedexNet content can be published through static hosting systems for broad public access.

Static hosting is suitable for documentation, generated species pages, software releases, public manifests, checksums, project information, small data packages, and fallback content.

Large media collections, dynamic search, private records, databases, background jobs, and administrative services require separate infrastructure.

Static mirrors should use predictable paths, relative links, versioned assets, and minimal client-side dependencies.

Web Delivery

NGINX Deployment

NGINX can serve static releases, proxy dynamic services, terminate TLS, enforce request limits, cache public responses, and route traffic among SpeciedexNet components.

Separate locations can serve immutable release objects, public APIs, generated pages, documentation, media, and administrative services under different policies.

Directory listing, internal files, source archives, secrets, database files, backup files, and configuration data should not become publicly accessible through incorrect web-root placement.

Node operators should review MIME types, cache headers, compression, range requests, upload limits, proxy timeouts, access logs, and security headers.

Version Compatibility

Protocol and API Compatibility

SpeciedexNet services should advertise supported protocol, schema, API, release, and software versions.

Clients should reject or degrade gracefully when a remote node cannot satisfy the requested version or feature.

Changes that break compatibility should be documented, versioned, tested, and accompanied by migration or fallback guidance.

Long-lived archives should preserve the documentation and software required to interpret historical releases.

Interoperability

Open Formats and Standards

SpeciedexNet is intended to use open, documented, and broadly implementable formats where practical.

Releases may use structured JSON, JSON Lines, delimited files, geographic formats, archive packages, standard hashes, digital signatures, machine-readable manifests, and recognized biodiversity schemas.

Project-specific extensions should remain documented, versioned, and exportable.

Independent developers should be able to build compatible readers, mirrors, validators, and archival tools without relying upon undocumented private interfaces.

Network Coordination

Governance and Node Policy

SpeciedexNet nodes may be operated by the project, institutions, partners, researchers, archives, or independent organizations.

Operators retain control over their infrastructure but must not misrepresent locally modified content as an unchanged official release.

Official node listings may require identity verification, minimum service standards, current security practices, accurate release labeling, and compliance with source licenses.

A node may be removed from approved listings when it distributes altered content without disclosure, exposes restricted data, remains dangerously outdated, or presents a material security risk.

Open Infrastructure

Open-Source Network Software

SpeciedexNet is intended to use open-source software, reproducible configuration, documented protocols, and auditable verification tools.

Open software allows institutions to operate mirrors independently, review security controls, preserve data, and adapt services to regional needs.

Public source code does not eliminate the need for secure deployment, maintenance, scientific review, key protection, and responsible administration.

Third-party components remain subject to their respective licenses, update practices, and security limitations.

Scientific and Technical Notice

SpeciedexNet is an evolving research, preservation, and distributed-software project.

Node availability, release status, compatibility, geographic coverage, search coverage, mirror accuracy, and service performance may vary.

Data may be incomplete, provisional, disputed, stale, locally extended, restricted, or unavailable from a particular node.

A verified release proves that content matches the referenced package or signed manifest. It does not prove that every scientific record within the release is complete or correct.

Users should verify important scientific information through current primary literature, recognized authorities, source databases, qualified specialists, and official agencies.
